## 0. 背景与约束(已与用户确认)

| 维度 | 结论 |
|------|------|
| 射程 | 延伸到生产/长期演进(不止于根 spec 的 P1/P2) |
| 终点画像 | **开源标杆作品**:架构完整 + 评测可复现 + 叙事漂亮,持续迭代的资产 |
| 演进主线 | 可复现工程化 + 检索技术深化 + Agent 能力拓展(不扩语言/领域) |
| 算力 | 主力单机 M 系列 32GB + 按需租云做一次性重活 |
| 验收口径 | **外部可复现**:陌生人 clone 仓库 → 一条命令跑出 ViDoRe 评测,结果与 README 报数在容忍带内 |
| 编排策略 | **A 评测可复现骨架为先**:可复现骨架贯穿全程,检索深化与 Agent 拓展作为"挂在骨架上、被评测验证才合入"的迭代 |

根 spec 的 P1(检索 MVP)、P2(GraphRAG+ReACT)作为**地基阶段**;长期演进在 P2 完成后展开,分 E1 检索深化期 / E2 Agent 演进期 / E3 标杆收口期。地基阶段从第一天就嵌可复现骨架。

## 1. 总体形态与编排原则

### 1.1 结构

```
        ┌──────────────────────────────────────────┐
        │   可复现骨架(Repro Spine)——贯穿全程        │
        │  CI 消融 · 索引/模型版本化 · 一键复现脚本    │
        │  容器化 demo · leaderboard 自动更新          │
        └──────────────────────────────────────────┘
               ▲          ▲          ▲
        ┌──────┘    ┌─────┘    └──────┘
   检索深化主线  Agent 拓展主线  (评测口径守住)
   SPLADE/HyDE/  阶段2 ReACT→
   查询改写/      多跳/自反思/
   多跳重排       工具收窄演进
```

### 1.2 三条编排纪律(贯穿所有阶段,不可破)

1. **合入即评测**:任何检索/Agent 新能力,必须先在 ViDoRe 消融里证明有增量(或至少不退步)才合入主分支。没增量的能力不进主线,只留实验分支。
2. **复现优先于峰值**:遇到"数字更好看但别人复现不出"的取舍,一律选复现。与验收口径硬绑定。
3. **单机可跑是底线,租云做重活**:所有"主线合入物"必须在单机 M 系列可复现;只在"全量重编码/模型升级重训/CI 评测"这类一次性重活按需租云,产物(索引/权重/评测报告)回本地长期可用。

## 2. 可复现骨架(Repro Spine)

骨架是 Strategy A 的命脉,落地成 5 个机制。

### 2.1 一键复现脚本(最高优先级,验收口径直接挂这)

```
make eval-vidore      # clone 仓库后第一条命令:跑 ViDoRe industrial 消融
make eval-full        # 全 8 子集(按需租云或本地长跑)
make demo             # 启动在线 demo
make fetch-indexes    # 首次拉取预编码索引(Release artifact)
```

- 入口收敛到 `Makefile` / `justfile`,底下转 `python -m evaluation.*`。
- `make eval-vidore` 产出 `results/<commit-sha>/vidore_industrial.json` + 消融表 markdown;README 报数必须能被这条命令复现到 **复现容忍带 ±0.5 NDCG@10**(见 §6 风险1:漂移时放宽到 ±1.0 但须显式标注)。

### 2.2 索引 / 模型版本化

- 索引(FAISS ColPali 多向量 + pgvector BGE 单向量)按 `<model_id>-<corpus_version>-<sha>` 命名,落 `indexes/`(大文件已 gitignore,走 Release artifact)。
- 模型版本用 `config/models.yaml` 钉死:ColPali=`vidore/colpali-v1.3`、BGE=`BAAI/bge-large-en-v1.5`、reranker=`BAAI/bge-reranker-large`、LLM=`ollama/qwen2:7b`。换版本 = 改 yaml + 重编码 + 重评测,**不允许代码里硬编码模型名**。

### 2.3 CI 跑消融(GitHub Actions)

- 每次 PR 跑 `make eval-vidore`(industrial 子集 ~5k 页,租云 GPU runner 或本地 self-hosted)。
- CI 产出消融对比:本 PR vs main 的 NDCG@10 / Recall@5 / MRR diff。**回归 > 容忍带则 CI 红**,强制纪律 1 自动生效。
- 重活(全量重编码)走 **scheduled job**(夜跑或手动 dispatch),不卡 PR。

### 2.4 Demo 服务

- API 服务 + 前端（可选），带一份预编码好的 Demo 知识库索引(350 份 ~3.5GB,放 Release artifact 挂载)。
- 这是"外部可复现"的第二张脸:不只跑得出评测,还能点得到 demo。

### 2.5 Leaderboard 自动更新

- CI 评测完后,结果自动写进 `README.md` 的 leaderboard 表 + `docs/leaderboard-history.csv`(历史趋势线)。
- 与 ViDoRe 公开 leaderboard 对比列(同 corpus、同切分)并排展示,不私改 corpus。

### 2.6 索引分发方式(已决)

索引大文件(FAISS ~3.5GB Demo / ~12GB 全量)走 **GitHub Release artifact + `make fetch-indexes` 首次拉取并缓存**,不走 Git LFS。理由:仓库保持轻量,符合"clone 后一条命令跑起来"的体感;LFS 在大文件频繁重编码时笨重。

## 3. 检索技术深化主线

在 P1 三路检索(BM25+Dense+Visual)+RRF+rerank 地基上长期深化。每个候选能力都是"先消融、有增量才合入"。

### 3.1 候选能力池(按收益预期排序,非全做)

| # | 能力 | 解决地基的什么盲区 | 预期增量 | 风险/成本 |
|---|------|-------------------|---------|-----------|
| 1 | 查询改写/扩展(HyDE / LLM 改写) | 设备编号、缩写、口语化查询的召回盲区 | 中高 | 低,纯推理期 |
| 2 | 稀疏向量路(SPLADE / BGE-M3 sparse)替换或增强 BM25 | BM25 词频盲区,术语加权更学得会 | 中 | 中,需训/加载稀疏模型 |
| 3 | 多跳重排(Cross-encoder + 多跳证据链) | 多跳 QA(第二阶段 GraphRAG 之后) | 中 | 中,依赖 graph 路就绪 |
| 4 | ColQwen / ColViDoRe 升级替换 ColPali-v1.3 | 模型迭代带来原生多向量增益 | 中高 | 高,需重编码全量 |
| 5 | 混合表示(Dense + Sparse 统一向量空间,如 BGE-M3) | 路数太多融合权重难调,合并降复杂度 | 中 | 中,重构融合层 |

**明确不做**(防 scope 蔓延):图块 CLIP 检索(根 spec §3.2 已否决)、Web 检索接入(根 spec §7.2 已否决)、跨语言检索(主线未选)。

### 3.2 接入纪律(与 §1.2 三纪律对齐)

每个候选能力的落地流程固定为 **"实验分支 → 消融对比 → 决议"** 三步:

1. **实验分支** `exp/retrieval/<capability>`:实现 + 加进 `FusionStrategy` 注册表(不污染主分支)。
2. **消融对比**:CI 跑 industrial 子集,与本分支基线比 NDCG@10。必须报两组——路由增量组(该能力 vs 地基三路)和重排增量组(若有 rerank 变动),与根 spec §5.3 纪律一致。
3. **决议**:增量 ≥ **合入增量门槛**(+0.5 NDCG@10,与 §2.1 的"复现容忍带"是两个概念:前者是新能力相对基线的最低增益,后者是别人复现你报数时的允许偏差)且不拖累其他路 → 合入主线;否则留实验分支或废弃,**废弃也要在 `docs/experiments.md` 留档**(为什么没增量),避免后人重复试。

### 3.3 与融合层的关系

- 地基是 `RRFFusion`(非参数),深化能力**先以独立路接入 RRF**(新增第 N 路,RRF 天然容纳多路)。
- 当路数 ≥4 且 RRF 路数敏感时,切 `ConvexFusion`(根 spec §7.1 已预留接口),权重由全量消融定标——**不在调用点 if/else 硬切**,长期沿用。
- 路数超过 5 后考虑 §3.1 #5 混合表示做路数收敛,避免融合层爆炸。

## 4. Agent 能力拓展主线

在 P2 的 ReACT Agent(工具集收窄到 `knowledge_search` / `query_knowledge_graph` / `query_refiner`)地基上长期演进。Agent 是全作品最难复现、最难评估的部分,所以这条主线的编排纪律比检索更严:**先建 Agent 评测,再扩能力**。

### 4.1 先决条件:Agent 评测基础设施(P2 完成时同步建)

否则后面所有 Agent 演进都是"看起来更聪明"的玄学,过不了"外部可复现"验收。

| 评测层 | 测什么 | 用什么 |
|--------|--------|--------|
| 轨迹正确率 | Thought/Action/Observation 链是否对(不只看最终答) | 自建 50 QA 的人工标注轨迹 + LLM judge 异家族复评 |
| 工具调用准确率 | 该不该调工具、调哪个、参数对不对 | 拒答集 20 条(不该调) + 多跳集(该调 graph) |
| 端到端答对率 | 最终答案对不对 | 50 QA + 20 拒答,与根 spec §5.1 三层对齐 |

> Judge 用 Ollama 同模型自评的偏好风险已在根 spec §5.1 承认;Agent 评测**从第一天就切 DeepSeek-V3 / GPT-4o-mini 异家族 judge**,不复用本地 qwen2:7b 自评。

### 4.2 候选能力池(按依赖与风险排序)

| # | 能力 | 依赖 | 解决什么 | 风险 |
|---|------|------|---------|------|
| 1 | 多跳查询改写(query_refiner 长期化) | P2 query_refiner | "上述比率"这类指代在多轮内显式解析 | 低 |
| 2 | 自反思 / Self-Refine | P2 ReACT | 答案不达标时回退重检索 | 中,易死循环 |
| 3 | 工具自适应调度(路由器决定调哪几把工具) | §3 检索路数变多后 | 不该全路都跑,按查询类型裁剪 | 中 |
| 4 | 多智能体协作(规划者+检索者+校验者) | 1/2/3 稳定 | 复杂查询拆解 | 高,评估爆炸 |
| 5 | 轨迹可解释可视化 | 前端 | 复现叙事的"看得见"层 | 低,但工程量大 |

### 4.3 接入纪律(比 §3.2 更严)

Agent 能力的合入门槛**高于**检索能力,因为 Agent 评测噪声大:

1. **实验分支** `exp/agent/<capability>`。
2. **三层评测都过**:轨迹正确率 + 工具调用准确率 + 端到端答对率,**三层都不能退步**才合入。只端到端涨、轨迹退步的不算成功(可能是运气)。
3. **成本约束**:新能力不能让单 query 的平均工具调用次数翻倍以上——Agent 演进不是"调越多越好",根 spec §7.2 工具集收窄的纪律长期延续。
4. **复现硬约束**:Agent 轨迹必须可回放(记录每步 Thought/Action/Observation + 用的检索结果 hash),CI 能 replay 验证。做不到可回放的能力不合入。

### 4.4 明确不做

- **不引入 `web_search`**(根 spec §7.2 已否决,长期延续):外网接入的 prompt injection / 来源可信度 / 去重问题超出作品边界。
- **不做通用 Agent 框架**(不把 PrismRAG 包装成 LangChain 替代品):它是"私有知识库 RAG + 受控 Agent",不是 Agent 平台。

## 5. 阶段路线图与里程碑

### 5.1 总览

```
P1 检索 MVP ─→ P2 GraphRAG+ReACT ─→ E1 检索深化期 ─→ E2 Agent 演进期 ─→ E3 标杆收口期
   地基阶段( 根 spec 已有 )        ↑              ↑              ↑
                            主线在此之上挂载展开
```

地基阶段(P1/P2)从第一天就嵌可复现骨架,不是 P2 后才补——这是 §1.2 纪律 1 的落地。

### 5.2 各段定义与验收点

| 阶段 | 内容 | 验收点(过了才进下一段) |
|------|------|------------------------|
| P1 检索 MVP | ingestion + 三路检索 + RRF + rerank + ViDoRe industrial + 20 条拒答 sanity | `make eval-vidore` 跑通、消融表成型、README 报数与复现容忍带内 |
| P2 GraphRAG+ReACT | Neo4j 图谱 + ReACT Agent + ConvexFusion + 端到端 50QA/20拒答 | Agent 轨迹可回放、三层评测基线建立(§4.1) |
| E1 检索深化期 | §3.1 候选能力池按序实验:#1 查询改写 → #2 稀疏向量 → #5 混合表示;每条走"实验分支→消融→决议" | 至少 2 条能力合入主线且 leaderboard 数字上涨;`docs/experiments.md` 废弃留档完整 |
| E2 Agent 演进期 | §4.2 候选池:#1 多跳改写 → #2 自反思 → #3 工具自适应;每条走三层评测门槛 | 至少 1 条能力合入;轨迹回放 CI 稳定;工具调用次数未翻倍 |
| E3 标杆收口期 | §3.1 #4 ColQwen 升级重编码 + 全量 ViDoRe leaderboard 提交 + 容器化 demo 打磨 + 技术博客/案例卡 5 个 | 外部陌生人 clone → 一条命令跑出评测 ±容忍带 → demo 可点 → leaderboard 有名 |

### 5.3 阶段切换的硬门(防抢跑)

每段进下一段的"硬门"是**机制就位**,不是"差不多该往下走了":

- **P1→P2 门**:可复现骨架(§2.1/2.2/2.3)就位。没骨架不进 P2,否则 P2 的 Agent 评测没法回放。
- **P2→E1 门**:Agent 三层评测基线(§4.1)就位。没评测基线不扩 Agent 能力,否则 §4 候选池全是玄学。
- **E1→E2 门**:检索深化至少 2 条合入且 leaderboard 涨。检索没站稳不扩 Agent,否则地基不稳。
- **E2→E3 门**:Agent 至少 1 条合入且轨迹回放 CI 稳定。没可回放不收口,否则验收口径("外部可复现")破。

### 5.4 算力租云的触发点

按 §1.2 纪律 3,只在以下一次性重活租云,产物回本地:

| 触发点 | 租云做什么 | 产物 |
|--------|-----------|------|
| P1 完成时 | 全量 ViDoRe 8 子集 ColPali 编码(~4.5h) | `indexes/colpali-v1.3-vidore-full-<sha>` |
| E1 #4 升级时 | ColQwen 全量重编码 | 新版索引 + 评测报告 |
| E3 收口时 | 全量 ViDoRe 评测 + leaderboard 提交 | leaderboard 名次 + 历史趋势 csv |

## 6. 风险与降级

| # | 风险 | 触发信号 | 降级预案 |
|---|------|---------|---------|
| 1 | 复现漂移——别人 clone 后数字对不上 | CI 复现 ≠ 本地数字 / issue 报"跑不出来" | 强制钉死 `config/models.yaml` + 索引 sha + 随机种子;容忍带放宽到 ±1.0 但必须在 README 显式标注;仍漂移则回退到最近一次可复现 commit |
| 2 | 单机算力撑不住——E1/E2 索引膨胀超 32GB | OOM / 编码超 8h | 索引换 FAISS HNSW32+PQ 量化(根 spec §3.4 备选转正);全量 ViDoRe 永久走租云,本地只载 Demo 子集;E3 收口时若仍撑不住,leaderboard 只报 industrial 子集不报全量 |
| 3 | Agent 评测噪声大——三层评测抖动,合入门槛形同虚设 | 同一 commit 两次跑分 diff > 5% | 评测集从 50→100 QA 扩容降方差;judge 从单模型切多模型投票;门槛从"不能退步"放宽到"置信区间不重叠退步" |
| 4 | 检索深化全无增量——§3 候选能力都过不了容忍带 | E1 期间 5 条候选全废弃 | 不强求合入,把 `docs/experiments.md` 废弃留档本身作为 E1 产出("试过、为什么没增量"也是叙事);E1 直接收口进 E2,不卡路线 |
| 5 | 租云成本失控——重编码频次变高 | 月云费超预算(自定阈值) | 模型升级频次限死:ColPali/ColQwen 类大重构 ≤ 每季度一次;中间迭代只跑 industrial 子集不全量 |
| 6 | scope 蔓延到通用 Agent 平台——想加 web_search / 多租户 / 插件市场 | 出现 §4.4 明确不做的需求 | 硬否决,引用根 spec §7.2 与本设计 §4.4;若强烈要,另开仓库不污染 PrismRAG |

**贯穿性兜底**:任何阶段若长期卡住(>预期工期 2 倍),触发"最小可交付收口"——把当前已合入物打包成 E3 形态(一键评测 + demo + leaderboard 提交),作为作品 v1.0 发布,后续作 v1.x 迭代。**开源标杆作品不追求"全做完",追求"已发布部分都站得住"。**
